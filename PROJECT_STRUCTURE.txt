================================================================================
KEY ARCHITECTURAL PATTERNS
================================================================================

1. HEXAGONAL ARCHITECTURE (Ports & Adapters)
   - Domain at the center (no external dependencies)
   - Application layer defines ports (interfaces)
   - Infrastructure/API layers are adapters

2. DOMAIN-DRIVEN DESIGN (DDD)
   - Bounded Contexts: cars, owners, agents, registration
   - Aggregates: Vehicle, Owner, Agent, Registration
   - Value Objects: VehicleId, Vin, OwnerId, FullName, etc.
   - Domain Events: (Can be added for event-driven communication)

3. ANTI-CORRUPTION LAYER (ACL)
   - Registration module uses ACL adapters to communicate with other modules
   - Prevents tight coupling between bounded contexts
   - Adapters: VehicleEligibilityAdapter, OwnerEligibilityAdapter, AgentEligibilityAdapter

4. REPOSITORY PATTERN
   - Port: *RepositoryPort interfaces in application layer
   - Adapter: Jpa*RepositoryAdapter in infrastructure layer
   - Spring Data JPA: SpringData*Repository interfaces

5. MAPPER PATTERN
   - Domain <-> DTO mapping in API layer
   - Keeps domain models isolated from HTTP concerns

6. EXCEPTION HANDLING
   - Domain exceptions: Business rule violations
   - Application exceptions: Use case failures
   - Global handler: shared/web/GlobalExceptionHandler
   - Module-specific handlers: *ExceptionHandler in each API layer

================================================================================
PROJECT CONVENTIONS
================================================================================

1. Package naming: com.champsoft.vrms.modules.{module}.{layer}
2. Class naming:
   - Entities: {Name}JpaEntity
   - Repositories: SpringData{Name}Repository, Jpa{Name}RepositoryAdapter
   - Services: {Name}CrudService, {Name}EligibilityService
   - Controllers: {Name}Controller
   - DTOs: {Action}{Name}Request/Response
   - Mappers: {Name}ApiMapper
   - Exceptions: {Description}Exception

3. Layer responsibilities:
   - Domain: Business logic, no framework dependencies
   - Application: Use cases, orchestration, port definitions
   - Infrastructure: Database, external services, adapters
   - API: HTTP layer, DTOs, controllers, exception handling

4. Module independence:
   - Each module is self-contained
   - Cross-module communication through ports/adapters (ACL)
   - No direct dependencies between modules

================================================================================
NOTES
================================================================================

- The project uses Spring Modulith principles for modular monolith architecture
- Database migrations are managed by Flyway (version-controlled SQL scripts)
- H2 is used for development; PostgreSQL for production
- All JPA entities use public fields for simplicity (can be refactored)
- Value Objects enforce domain invariants through validation
- The registration module orchestrates cross-module operations
- ACL adapters ensure loose coupling between bounded contexts
